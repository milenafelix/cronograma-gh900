## Flashcards

- Diferença entre git fetch e git pull
    
    O ‘fetch’ apenas baixa os dados do remoto para as branches de rastreamento. O ‘pull’ executa o ‘fetch’ e imediatamente tenta fazer o ‘merge’ na branch atual.
    
- Função do GitHub Dependabot
    
    Monitorar vulnerabilidades em dependências e abrir Pull Requests automaticamente para atualizá-las para versões seguras.
    
- GitHub Discussions vs. GitHub Issues
    
    Discussions: Brainstorming, perguntas e conversas informais.
    
    Inssues: Tarefas acionáveis, bugs e rastreamento de progresso.
    
- Comando para criar um repositório local novo
    
    git init
    
- Finalidade do comando git clone
    
    Criar uma cópia local completa de um repositório que já existe no servidor remoto
    
- O que é um ‘Fork’ no GitHub?
    
    Uma cópia de um repositório que vive na sua conta, permitindo que você altere o código sem afetar o projeto original
    
- Staging Arrea (Área de Preparação)
    
    O local onde os arquivos são colocados (via git add) antes de serem confirmados definitivamente no histórico.
    
- Comando para registrar mudanças no histórico local.
    
    git commit -m “mensagem”
    
- Finalidade das Branchs Protection Rules
    
    Garantir a integridade do código, impedindo pushes diretos, exigindo revisões ou testes automatizados antes do merge.
    
- O que é InnerSource?
    
    Uso de práticas de desenvolvimento de código aberto (colaboração aberta, transparência) dentro de uma organização privada.
    
- Papel do arquivo CODEOWNERS
    
    Definir indivíduos ou equipes responsáveis por revisar alterações em partes específicas de um repositório.
    
- GitHub Actions: Gatilho ‘workflow_dispatch
    
    Permite que um fluxo de trabalho seja iniciado manualmente através da interface do GitHub ou API.
    
- Flluxo de trabalho básico do Git (Etapas)
    1. Criar branch
    2. Commit alterações
    3. Abrir Pull Request
    4. Revisar e Mesclar
- Melhor forma de gerenciar permissões em organizações
    
    Através de Equipes (Teams), atribuindo níveis de acesso ao grupo em vez de indivíduos separadamente.
    
- O que é um Pull Request?
    
    Uma proposta formal para mesclar alterações deuma branch em outra, permitindo revisão de código e discussão.
    
- Gatilhos comuns do Github Actions
    - push
    - pull_request
    - schedule (cron)
    - workflow_dispatch
- Diferença entre repositório Privado e Interno
    - Privado: Apenas pessoas convidadas veem
    - Interno: Todos os membros da organização corporativa veem
- GitHub Projects: Visualização Kanban
    
    Organiza tarefas em colunas (ex: To Do, In Progress, Done) para gerenciar o fluxo de trabalho visualmente
    
- Tratamento de Vulnerabilidade em PR
    
    Solicitar alterações e fornecer feedback detalhado; nunca aprovar código inseguro para corrigir depois.
    
- Três pilares do DevOps
    - Cultura de colaboração
    - Automação (CI/CD)
    - Feedback contínuo (monitoramento).
- Comando para ver o status atual dos arquivos
    
    git status
    
- Uso de Marcos (Milestones)
    
    Agrupar Issues e Pull Requests para acompanhar o progresso em direção a uma meta ou data específica.
    
- GitHub Secrets
    
    Variáveis criptografadas que armazenam senhas e tokens de API com segurança para uso em actions
    
- Efeito do comando git push
    
    Envia os commits realizados no repositório local para repositório remoto no GitHub.
    
- Draft Pull Request
    
    Um estado de PR que sinaliza trabalho em andamento e impede a mesclagem até que seja marcado como pronto para revisão.

## Prompt para o Gemini criar simulado, flashcards e guia de estudo (modo raciocínio ou pro)

[Alimentar a IA](Alimentar%20a%20IA%20311c330adf8380c684cbc937da52b183.md)

Você é um instrutor certificado em GitHub Foundations e especialista em versionamento de código.

Crie um simulado completo no estilo da prova oficial GitHub Foundations, contendo:

- 30 questões de múltipla escolha
- 4 alternativas por questão (A, B, C, D)
- Apenas uma alternativa correta
- Nível iniciante a intermediário

Requisitos obrigatórios de qualidade das questões:

- Todas as alternativas devem ter tamanho semelhante (quantidade de palavras parecida).
- Todas devem ter o mesmo nível de detalhamento técnico.
- Nenhuma alternativa pode ser claramente mais longa ou mais curta que as outras.
- Evite respostas obviamente erradas ou absurdas.
- Evite termos como “sempre”, “nunca”, “apenas se” que denunciem a alternativa correta.
- As alternativas devem ser plausíveis e tecnicamente coerentes.
- Não deixar pistas estruturais que facilitem a identificação da resposta correta.

Os temas devem incluir:

- Conceitos básicos de Git e GitHub
- Repositórios, branches e commits
- Pull requests e code review
- Issues e GitHub Projects
- GitHub Actions (noções básicas)
- Colaboração em times
- Segurança básica e permissões

Ao final, forneça:

- Gabarito separado das questões
- Explicação breve e técnica de cada resposta correta
- Justificativa objetiva do porquê as outras alternativas estão incorretas
